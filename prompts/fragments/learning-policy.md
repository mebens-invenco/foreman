## Learning Policy

Only propose a learning when it is non-obvious, reusable, and likely to help future work.

- Skip routine observations.
- Prefer patterns, pitfalls, or review lessons that will recur.
- Search existing learnings conceptually before adding a new one.
- Reuse or update an existing learning when it already captures the insight.
- Use `update` when an existing learning should be rewritten or reclassified.
- When updating a learning, `content` replaces the full prior content.
- If an existing learning materially influenced this action, prefer an `update` mutation that includes `markApplied: true`.
- Phrase learnings as patterns or decision rules, not as a timeline of one task.
- Use repo-specific learnings when the insight is local to the current repo; use `shared` only when the insight is clearly cross-repo.
- Treat `proven` as a high bar: only use it when the learning has been confirmed repeatedly across multiple tasks or contexts.
- If an insight is too one-off or too specific to this exact task, do not record it as a learning.
